# App Store Listing Creator

Build an evidence-based listing package for Google Play, Apple App Store, or both. Produce actual images when an image-generation capability is available and authorized; otherwise produce generation-ready creative briefs and prompts.

## Operating rules

- Follow the user's language for deliverables unless a target locale requires another language.
- Inspect supplied product evidence and existing assets before proposing copy or visuals.
- Verify current dimensions, file formats, character limits, field availability, and review policies against official Apple or Google documentation before giving exact submission constraints.
- Never invent awards, rankings, adoption numbers, testimonials, certifications, privacy claims, or capabilities.
- Do not promise store approval or conversion improvement.
- Read [evidence and verification](../shared-references/evidence-and-verification.md) before evaluating claims or reporting expected outcomes.

## Workflow

### 1. Establish the listing matrix

Identify:

- stores, app type, package/bundle identifiers, release stage, and target date;
- target countries, languages, devices, orientations, and form factors;
- primary audience, job to be done, differentiators, pricing, and required disclosures;
- available build, screenshots, icon/brand files, design tokens, analytics, reviews, and approved product facts.

Ask only for missing information that would materially change the result. If safe, continue with clearly labeled assumptions.

### 2. Audit existing material

For an existing listing, record each field and asset before editing. Evaluate message hierarchy, consistency with the product, first-impression clarity, legibility at store scale, localization quality, unsupported claims, duplicated frames, and gaps between screenshots and the live experience.

Separate observed evidence from hypotheses. Do not infer conversion causes from a single aggregate metric.

### 3. Verify current platform constraints

Use current official store documentation. Record the source and access date for volatile rules. Build a platform/locale/asset matrix containing required and optional fields, exact limits, formats, device sets, and policy-sensitive content.

Do not reuse an old remembered limit when official documentation is accessible.

### 4. Define positioning

Write:

1. one primary audience statement;
2. one core promise supported by product evidence;
3. three to five proof-backed value pillars;
4. a message hierarchy for the first impression, evaluation, and trust stages;
5. keyword themes mapped to real intent rather than repeated mechanically.

Resolve conflicting messages before producing assets.

### 5. Create the icon direction

Produce two or three meaningfully different concepts. For each include the central metaphor, silhouette, composition, palette, depth/material treatment, small-size behavior, platform adaptation, brand linkage, risks, and a generation-ready prompt.

Avoid text, screenshots, tiny detail, misleading badges, and direct imitation of a competitor. Test recognition at small size, monochrome/background variants when relevant, and contrast against common store surfaces.

If the user authorizes image creation and an image tool exists, generate the selected concept only after the direction is clear. Preserve editable source or prompt details when possible.

### 6. Build the screenshot narrative

Treat screenshots as a sequence, not isolated posters:

- Frame 1: instantly identify the product outcome.
- Frames 2–3: show the strongest differentiators with real UI evidence.
- Middle frames: explain workflow, depth, or trust.
- Final frame: reinforce the next action or ecosystem value without unsupported urgency.

For every frame specify objective, headline, optional support line, source screen/state, crop, device treatment, layout, background, visual emphasis, localization risk, accessibility/legibility note, and generation or compositing prompt. Never fabricate UI that implies unavailable functionality.

### 7. Write store copy

Create only fields supported by the verified store matrix. Keep the opening specific, readable, and outcome-led. Explain features through user value, then include necessary trust, subscription, compatibility, or legal details.

Map keywords naturally to relevant fields. Do not stuff, repeat irrelevant terms, use competitor trademarks deceptively, or claim superlatives without evidence. Preserve mandatory legal and privacy language exactly when supplied.

### 8. Localize deliberately

For each locale, identify literal-translation risks, cultural or legal sensitivities, text expansion, typography, screenshots requiring replacement, keyword intent differences, and claims needing local evidence. Prefer transcreation guided by the positioning over word-for-word translation.

### 9. Assemble and verify

Read [deliverables](references/deliverables.md) and produce its complete package. Check every fact against evidence, every asset against the current platform matrix, copy against verified limits, screenshots against the real product, and locale variants for truncation and meaning.

When experimentation is possible, propose one-variable hypotheses with success and guardrail metrics. Label them as experiments, not predicted wins.

## Mutation boundary

Creating briefs, copy, and an audit is read-only. Editing design files, generating images, changing store-console content, or uploading assets requires explicit user authorization and the relevant access. Never submit or publish a listing merely because the user requested an audit or creation package.
